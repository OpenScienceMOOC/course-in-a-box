<h2><a id="user-content-set-up-a-github-repository-" class="anchor" aria-hidden="true" href="#set-up-a-github-repository-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Set up a GitHub repository <a name="user-content-Setup"></a></h2>
<blockquote>
<p><strong>Pro-tip</strong>: Make sure to include a LICENSE and README file in your repository. This will indicate to people the purpose of the project, and how they can engage with it in the future.</p>
</blockquote>
<p>Find out how to set up a GitHub repository in this other guide <a href="/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/Task_1.md">Task 1: Building a GitHub repository</a> which is also part of 'Module 5: Open Research Software and Open Source'.</p>
<h2><a id="user-content-choose-your-github-repository-" class="anchor" aria-hidden="true" href="#choose-your-github-repository-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Choose your GitHub repository <a name="user-content-Choose"></a></h2>
<p>Once on your GitHub project listings page at <a href="https://github.com">github.com</a> head to the 'Repositories' tab. Select which repository you would like to archive, and open it up.</p>
<br>
<h2><a id="user-content-login-to-zenodo-" class="anchor" aria-hidden="true" href="#login-to-zenodo-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Login to Zenodo <a name="user-content-Login"></a></h2>
<p>Now head over to <a href="https://zenodo.org" rel="nofollow">zenodo.org</a>. <a href="https://zenodo.org/" rel="nofollow">Zenodo</a> is a platform where you can permanently archive your code and other project elements. Zenodo does this by assigning projects a Digital Object Identifier (DOI), which also helps to make the work more citable. This is different to GitHub, which acts as a place where the actual work on a project takes place, rather than long-term archiving of it. At GitHub, content can be modified, deleted, rewritten, and irreversibly changed, which makes it a bit concerning to be used for longer lasting referencing purposes. Zenodo offers more security and permanence for research outputs.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/zenodo.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/zenodo.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Sign up for Zenodo</i></p>
<br>
<p>If you already have a Zenodo account, this is easy. If not, follow the steps to create one — you can even login using your GitHub account or ORCID profile to make things simpler, as Zenodo has a built in integration for it. This might be easier than creating yet another research account and profile.</p>
<br>
<h2><a id="user-content-authorise-github-to-connect-with-zenodo-" class="anchor" aria-hidden="true" href="#authorise-github-to-connect-with-zenodo-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Authorise GitHub to connect with Zenodo <a name="user-content-Authorise"></a></h2>
<p>On the Zenodo website authorise it to connect to your GitHub account in the '<a href="https://zenodo.org/account/settings/github/" rel="nofollow">Using GitHub</a>' section. Here, Zenodo will redirect you to GitHub to ask for permissions to use '<a href="https://developer.github.com/webhooks/">webhooks</a>' on your repositories. You want to authorise Zenodo here with the permissions it needs to form those links.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/zenodo_github.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/zenodo_github.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Authorize Zenodo to connect with GitHub</i></p>
<br>
<p>If you are trying to give Zenodo access to an organisational repository, you (or an administrator) will need to make sure that Zenodo is granted third party access permissions. GitHub will send an authorisation email that needs confirming. At this point, back in the settings of your repository on GitHub, you also need to make sure that the repository is set to 'public', not private.</p>
<br>
<h2><a id="user-content-select-the-repository-to-archive-" class="anchor" aria-hidden="true" href="#select-the-repository-to-archive-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Select the repository to archive <a name="user-content-Archive"></a></h2>
<p>If you have got this far, this means that Zenodo is now authorised to configure the repository webhooks that it needs to archive the repository and issue it a DOI. To do this, on the Zenodo website navigate to the <a href="https://zenodo.org/account/settings/github/" rel="nofollow">GitHub repository listing page</a> and simply click the 'on' button next to your repository.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/enabled_repos.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/enabled_repos.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Enable individual GitHub repositories to be preserved in Zenodo</i></p>
<br>
<h2><a id="user-content-check-repository-settings-" class="anchor" aria-hidden="true" href="#check-repository-settings-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Check repository settings <a name="user-content-Check"></a></h2>
<p>Now you have set up a new webhook between Zenodo and your repository. In GitHub, click on the settings for your repository, and the Webhooks tab on the left hand side menu. This should display the new Zenodo webhook configured to Zenodo. Note, it may take a little time for the webhook listing to show up.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/webhooks.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/webhooks.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Check that webhooks are enabled for your GitHub repository. Example here using the Open Scholarship Strategy</i></p>
<br>
<h2><a id="user-content-create-a-new-release-" class="anchor" aria-hidden="true" href="#create-a-new-release-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Create a new release <a name="user-content-Release"></a></h2>
<p>The first time you archive a repository is known as the 'first release'. Each time you create a new version of that repository and archive it, you create a new release. This can be tracked in the 'releases' tab for your repository on GitHub (top center).</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/first_release.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/first_release.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Check that the repository first release was successful. Example here using the Open Scholarship Strategy</i></p>
<br>
<p>For the first archived version of your repository, click 'Create a new release' back in Zenodo. Fill in the form and give some details as to what the release entails. For the first release, make sure to call it v1.0.0, as is standard practice.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/create_release.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/create_release.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Create a new release. Example here using the Open Scholarship Strategy, for which a first release already exists</i></p>
<br>
<p>Finally, click 'publish release', and your archive will be published and versioned on GitHub.</p>
<p>To view your release on Zenodo you need to visit the <a href="https://zenodo.org/deposit" rel="nofollow">Upload</a> tab. To finish the archiving a few more details are needed on Zenodo.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/upload_release.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/upload_release.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Check the new release has been uploaded. Example here shown using the Open Scholarship Strategy</i></p>
<br>
<h2><a id="user-content-getting-a-doi-" class="anchor" aria-hidden="true" href="#getting-a-doi-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Getting a DOI <a name="user-content-DOI"></a></h2>
<p>This is sometimes referred to as DOI 'minting', and requires a couple of extra bits of information about the repository on Zenodo. On Zenodo click the <a href="https://zenodo.org/deposit" rel="nofollow">Upload</a> tab in the main menu, and your newly uploaded repository should be there. Scroll down the page and fill in the extra information as needed, required fields are marked with a red asterisk, and then click 'Publish'.</p>
<p><strong>Note</strong>: Only after this extra information has been added will your DOI become live. It may also take a short time for the DOI to become active. Example DOI shown below (for the Open Scholarship Strategy again).</p>
<blockquote>
<p><strong>Pro-tip</strong>: Copy the URL for the DOI into the README file for your GitHub repo to make cross-linking even easier, as well as present a clear highlighted DOI badge for users to see and make use of your DOI. You only need to do this once with your first release DOI as it acts as a 'concept DOI' and is linked to all subsequent release DOIs.</p>
</blockquote>
<p><a href="https://doi.org/10.5281/zenodo.1323437" rel="nofollow"><img src="https://camo.githubusercontent.com/4d8878f6a7b0a57b272e3f17d48354221fc3c39a/68747470733a2f2f7a656e6f646f2e6f72672f62616467652f444f492f31302e353238312f7a656e6f646f2e313332333433372e737667" alt="DOI" data-canonical-src="https://zenodo.org/badge/DOI/10.5281/zenodo.1323437.svg" style="max-width:100%;"></a></p>
<p>The GitHub/Zenodo integration will now assign a DOI to each version/release of a project repository. This enables users to refer to and cite specific versions of projects. Also, the list of authors for the citation is automatically determined by the GitHub user account names used by the repository - this means no-one gets left out. Author details can be edited later on Zenodo. DOIs used in Zenodo are registered through the <a href="https://www.datacite.org/" rel="nofollow">DataCite</a> service.</p>
<blockquote>
<p><strong>Pro-tip</strong>: Create a 'human-readable' version of this citation in your project's README file. This will be helpful to researchers who might not be familiar with using DOIs to create citations, and make it easier for others to cite your software and acknowledge your work. An example of this could be:
Jon Tennant. (2018, July 30). Foundations for Open Scholarship Strategy Development: First formal release (Version 1.2). Zenodo. <a href="http://doi.org/10.5281/zenodo.1323437" rel="nofollow">http://doi.org/10.5281/zenodo.1323437</a></p>
</blockquote>
<p><strong>CONGRATULATIONS!!</strong></p>
<p>Your GitHub repository is now archived in Zenodo, and with a DOI that can be versioned to reflect updates to the repository version through time. You should be able to see details of this on the GitHub Zenodo page for your repository. This also means that your archived projects can get picked up by other indexing services and search engines that use DOIs too.</p>
<p>Providing a long-term archive and a DOI for your work is required for others to be able to properly cite it, as this provides basic citation metadata. For Open Science, it is important to be able to cite the software that you use in your research, and this integrated workflow enables that to happen, in line with best practices for research citation. Furthermore, this practice is important in elevating the standard of software (and related projects) to that of the standard of other research outputs.</p>
<blockquote>
<p><strong>Pro-tip</strong>: Is your research funded by an EU grant? Now you can directly connect your archived project to your grant by updating the grant section of the metadata on the project's Zenodo record. This massively helps to increase its discoverability!</p>
</blockquote>
<br>
<h2><a id="user-content-checklist-for-citing-your-project-" class="anchor" aria-hidden="true" href="#checklist-for-citing-your-project-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Checklist for citing your project <a name="user-content-Checklist"></a></h2>
<p>So now you have a sustainably archived GitHub repository in Zenodo that is ready to be re-used and cited! Before continuing, make sure that you have:</p>
<ul class="contains-task-list">
<li class="task-list-item"><input type="checkbox" id="" disabled="" class="task-list-item-checkbox"> Linked your GitHub project to Zenodo. If you see a complete copy of your GitHub repository in Zenodo then things are working.</li>
<li class="task-list-item"><input type="checkbox" id="" disabled="" class="task-list-item-checkbox"> Zenodo and GitHub integrated setup works nicely. For example have all the author names, and correct project title come across to Zenodo. If not, or if authors just have nicknames you can edit these details in Zenodo.</li>
<li class="task-list-item"><input type="checkbox" id="" disabled="" class="task-list-item-checkbox"> Project has a first release, with a DOI. You should have a DOI displayed on your projects Zenodo page. This first DOI is called the 'concept DOI' and is the master DOI linking to all subsequent release DOIs. Copy this DOI link and embed it in your GitHub projects README page. You're done!</li>
</ul>
