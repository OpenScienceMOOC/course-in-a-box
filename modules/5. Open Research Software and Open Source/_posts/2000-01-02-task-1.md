<h2><a id="user-content-getting-started-" class="anchor" aria-hidden="true" href="#getting-started-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Getting started <a name="user-content-Getting_started"></a></h2>
<p>A 'repository' is really just a fancy name for a project on GitHub. GitHub is a place online where you can manage projects, store files, and openly collaborate with others. This is all achieved by using version control to track projects as they progress. As such, GitHub is a powerful tool for both software and non-software projects.</p>
<p>One of the most important things to consider at this early stage is to think about how you want the wider community to interact with your project. As you are working in the open, you want to make sure others feel comfortable in accessing, viewing, and engaging with your work. Setting up a repository in a way that lowers the barriers to entry, and the fear of being an 'outsider' is the first step towards maintaining a successful project.</p>
<p align="center">
  <a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/octocat.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/octocat.png" width="150px" height="125px" style="max-width:100%;"></a>
</p>
<p align="center"><i>Octocat, GitHub's little mascot</i></p>
<br>
<h3><a id="user-content-setting-up-a-github-profile-" class="anchor" aria-hidden="true" href="#setting-up-a-github-profile-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Setting up a GitHub profile <a name="user-content-Profile"></a></h3>
<p>To set up a GitHub profile, simply head to the main page and click <a href="https://github.com/join">Sign Up for GitHub</a>. Here, you can create your personal account, with a username, email, and password as standard.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/github_signup.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/github_signup.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Sign up for GitHub</i></p>
<br>
<p>The next step is to set up a personal plan. For now, simply select the 'Unlimited public repositories for free' plan, unless you are concerned about privacy, in which case select the private plan. If you intend to set up a project for an organisation, you can select that option too.</p>
<br>
<h3><a id="user-content-the-github-language-" class="anchor" aria-hidden="true" href="#the-github-language-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>The GitHub language <a name="user-content-Language"></a></h3>
<p>This is possibly the most confusing and off-putting aspect of GitHub. Here are some of the most commonly used terms and their definitions:</p>
<ul>
<li><strong>Initialise</strong>: Create an empty repository.</li>
<li><strong>Checkout</strong>: Create a working copy of a local repository.</li>
<li><strong>Clone</strong>: Copy the repository into a local directory on your computer.</li>
<li><strong>Fork</strong>: Create a personal offshoot of a repository to work on it in parallel.</li>
<li><strong>Branch</strong>: An independent and parallel version of a repository. Changes do not affect the master branch.</li>
<li><strong>Master</strong>: The main and default branch for a repository.</li>
<li><strong>Clean</strong>: No commits pending on the branch.</li>
<li><strong>Stage</strong>: Add updates ready to be committed to a branch.</li>
<li><strong>Commit</strong>: A revision to a repository, like a versioned 'save' function.</li>
<li><strong>Commit message</strong>: A description of changes accompanying a commit.</li>
<li><strong>Check</strong>: A status check.</li>
<li><strong>Fetch</strong>: Nothing to do with dogs. Refers to getting the latest changes from an online repository without merging them.</li>
<li><strong>Index</strong>: The 'tree' which acts as a staging area.</li>
<li><strong>Working Directory</strong>: The 'tree' where the files are kept.</li>
<li><strong>Head</strong>: The 'tree' which indicates the last commits made.</li>
<li><strong>Push</strong>: Add committed changes to the head of your remote repository.</li>
<li><strong>Merge</strong>: Combining the changes made in one branch back with the master branch upon completion.</li>
<li><strong>Pull</strong>: Update your repository by fetching and merging the newest commits.</li>
<li><strong>Pull request</strong>: A request to merge an updated branch into the master branch.</li>
<li><strong>Issue</strong>: Suggested improvements, tasks, or questions related to a repository.</li>
</ul>
<p>Whew! Don't worry about memorising <em>all</em> of these for now. Like any new skill, familiarity comes with experience.</p>
<p>You can probably see how some of these are fairly similar to things like save, copy, paste - standard workflow operations, but adapted for a software management process. There are a <a href="https://mirrors.edge.kernel.org/pub/software/scm/git/docs/gitglossary.html" rel="nofollow">few more</a> too, but this should do for getting started.</p>
<p>If you are interested, most of these terms come from the underlying <a href="https://git-scm.com" rel="nofollow">Git system</a>. Git was built to allow developers to manage different versions of source code in a distributed manner, which is great. It has lots of features and the ability to do lots of complex stuff, written by a <a href="https://www.linuxfoundation.org/blog/10-years-of-git-an-interview-with-git-creator-linus-torvalds/" rel="nofollow">very clever guy</a>.  However, the <a href="https://xkcd.com/1597/" rel="nofollow">user interface was not designed with new users in mind</a>, so it can be hard to learn.</p>
<br>
<h3><a id="user-content-creating-a-new-repository-" class="anchor" aria-hidden="true" href="#creating-a-new-repository-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Creating a new repository <a name="user-content-Create_new"></a></h3>
<p>On your GitHub profile, click the 'Create new repository'. The first step is to create a name as the brand for your project. Ideally, it should be memorable and give some indication of what the project does.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/new_repo.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/new_repo.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Create a new repository</i></p>
<br>
<p>Make sure not to duplicate names, infringe upon other trademarks, or name it anything that could be considered to be offensive.</p>
<br>
<h2><a id="user-content-the-foundational-steps-" class="anchor" aria-hidden="true" href="#the-foundational-steps-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>The foundational steps <a name="user-content-Foundation"></a></h2>
<p>Any GitHub repository requires 4 key elements to get started and to begin developing a welcoming community:</p>
<ol>
<li>An Open Source license;</li>
<li>A README file;</li>
<li>Contributing guidelines; and</li>
<li>A Code of Conduct.</li>
</ol>
<p>These are critical aspects and best practices of any project for users to understand their legal rights, their expectations, the purpose of the project, and to improve the overall user experience.</p>
<p>All four of these files should be kept in the root directory for your project repository. It is convention to use markdown file formats (.md) for most of these files (though the license file is most often plain text (.txt)), and capitalise all file names. Instead of spaces in file names, make sure to use underscores _ .</p>
<p>So you should end up with a foundational file selection like this:</p>
<ol>
<li>LICENSE.md</li>
<li>README.md</li>
<li>CONTRIBUTING.md</li>
<li>CODE_OF_CONDUCT.md</li>
</ol>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/foundation.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/foundation.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>The basic repository structure</i></p>
<br>
<h3><a id="user-content-choosing-a-license-" class="anchor" aria-hidden="true" href="#choosing-a-license-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Choosing a license <a name="user-content-License"></a></h3>
<p>Choosing an appropriate license is what will differentiate your Open Source repository from publicly available software. While you are not obliged to choose a license, doing so guarantees that others will be able to modify, share, re-use, and build upon your project within a legal framework.</p>
<p>To start with, you want to check <a href="https://choosealicense.com/" rel="nofollow">Choose A License</a> to find a license that best suits your intentions for the repository.</p>
<p>The three primary ones to choose from are:</p>
<ul>
<li><strong>MIT License</strong>: A permissive license that lets people do whatever they want with your code as long as they provide appropriate attribution to you, and do not hold you liable.</li>
<li><strong>Apache License 2.0</strong>: Similar permissions to the MIT License, but also provides an express grant of patent rights from contributors to users.</li>
<li><strong>GNU General Public License (GPL) v3</strong>: A <a href="https://en.wikipedia.org/wiki/Copyleft" rel="nofollow">copyleft</a> license that requires anyone who redistributes your code, or a derivative work, to make the source available under the same terms as the original license; also provides an express grant of patent rights from contributors to users.</li>
</ul>
<p>Thankfully, when you start a new repository on GitHub, you are given the option to select an existing license from a drop-down menu.  You should always (with very few exceptions) use an existing license, since this is what potential users and contributors will see before they choose to use or contribute to your software.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/license.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/license.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Choosing an example license</i></p>
<br>
<p>If they don't have one you want, you can add one you like manually. To do this, simply click 'Create new file' in the repository, and copy and paste an existing license text in. Name the file something like 'LICENSE.txt' or 'LICENSE.md' to make it clear, and keep it in the main repository folder (i.e., the root). Make sure to add a clean commit message, and you're done!</p>
<br>
<h3><a id="user-content-creating-a-readme-file-" class="anchor" aria-hidden="true" href="#creating-a-readme-file-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Creating a README file <a name="user-content-Readme"></a></h3>
<p>When you initialise your new repository, there should be an option to do so with a README file. Just like Alice in Wonderland, these do exactly what they say - provide key information about the project. These are typically the first thing outside contributors will see when they come to your repository, so making them informative and welcoming is key.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/readme.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/readme.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Part of the README file for this module</i></p>
<br>
<p>The file will originally be in markdown (.md) format. This is a lightweight markup language with a plain text format. To learn some basic markdown, see <a href="https://github.com/adam-p/markdown-here/wiki/Markdown-Cheatsheet">this cheatsheet</a>. But for now, we can just use plain text.</p>
<p>There are several things you will want to include in your README file:</p>
<ul>
<li>What is this project about and what does it do.</li>
<li>Why should people care, and why is it useful.</li>
<li>How can someone get started contributing to the project.</li>
<li>Who can be contacted in case someone needs help.</li>
<li>A link to the license, contributing guidelines, and code of conduct.</li>
<li>A description of the project structure.</li>
<li>Who is involved, and what are their roles.</li>
<li>The current status of the project.</li>
</ul>
<p>Remember that not everyone coming to your project will be an expert, or understand what it is you are doing and why. Having a well-documented README file will enhance the user experience for people with a range of prior knowledge.</p>
<p>When the README file is included in the root directory, GitHub will automatically display this on the homepage of your repository. This means it is the first thing people will often see, so make it count!</p>
<blockquote>
<p><strong>Pro-tip</strong>: Later on as your project develops, you might want to add FAQs based on community feedback, or a tutorial to help users understand how your project works.</p>
</blockquote>
<br>
<h3><a id="user-content-creating-contributing-guidelines-" class="anchor" aria-hidden="true" href="#creating-contributing-guidelines-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Creating contributing guidelines <a name="user-content-Contributing"></a></h3>
<p>Contributing guidelines are designed to communicate to potential contributors a short guide on how to engage with your project and community. You want to make sure to be welcoming, and indicate that you are eager for participants to engage with your project. Whenever a participant opens a new pull request or creates a new issue, they will see a link to your contribution file.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/contributing.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/contributing.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Part of the CONTRIBUTING guidelines for this module</i></p>
<br>
<p>Sticking with the all caps file names, the next step is to create a CONTRIBUTING file. Click 'Create new file', and make sure to save it in markdown format as before. This file will tell other users how they can engage with and participate in your project. This is the first step towards establishing a community around your project, so make it engaging, concise, and informative.</p>
<p>The CONTRIBUTING file should include information on:</p>
<ul>
<li>What sort of contributions you are looking for.</li>
<li>How to suggest updates or new features.</li>
<li>How to interact with the project using GitHub's functions (e.g., the pull request protocol).</li>
<li>How to file a bug report or create an issue.</li>
<li>The ultimate goal, vision, or roadmap for the project.</li>
<li>How to contact those in charge of the project.</li>
<li>Links to any external documentation or websites.</li>
</ul>
<p>Here, you are essentially trying to encourage people to volunteer their time to advance your project. Make sure to be welcoming and friendly, and be precise about how people can engage. When writing this, make sure to think about it from the user perspective - how can you make their life easier when submitting pull requests and opening issues to make the whole project run more smoothly.</p>
<blockquote>
<p><strong>Pro-tip</strong>: Consider starting off with a short thank you note for people taking the time to consider contributing - they have clicked on the file to learn more after all! If there are other methods of recognition that you have in mind, make sure to include them in here too.</p>
</blockquote>
<br>
<h3><a id="user-content-creating-a-code-of-conduct-" class="anchor" aria-hidden="true" href="#creating-a-code-of-conduct-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Creating a Code of Conduct <a name="user-content-Conduct"></a></h3>
<p>A code of conduct is important for setting the ground rules for expected behaviour and participation for project contributors, and is an easily referenced document for showing that your project team takes constructive dialogues seriously. Therefore, it is a critical element for creating and maintaining a healthy community that engages in a constructive and productive manner within a positive social atmosphere.</p>
<p>A code of conduct not only provides expectations of behaviour, but also describes who those expectations apply to, when they apply, what to do should a violation of the code occur, and what the action items for this will be. As such, points of contact need to be made clear in the code of conduct. Typically, this should be in a private way such as an email address.</p>
<blockquote>
<p><strong>Pro-tip</strong>: In case a violation needs to be reported about the person who receives those reports, make sure to include an option to contact a secondary party.</p>
</blockquote>
<p>To add a code of conduct, you can create your own from scratch by adding a new markdown file, or use existing templates such as the <a href="https://contributor-covenant.org/" rel="nofollow">Contributor Covenant</a>. Name your file CODE_OF_CONDUCT.md, and make sure it is visible in the README file.</p>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/code_of_conduct.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/code_of_conduct.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>Part of the CODE OF CONDUCT file for this module, based on the Contributor Covenant</i></p>
<br>
<p>Making sure to enforce the code of conduct is important, as it shows that not only do you value the code, but you respect the influence that it has on your community. It is important to treat each member of the community with the respect, courtesy, and importance that they deserve. Should a violation occur, or a repeat offender makes consistent violations, it is best to refer to the <a href="https://opensource.guide/code-of-conduct/#enforcing-your-code-of-conduct" rel="nofollow">Open Source Guide</a> to see how to enforce the code of conduct.</p>
<br>
<h3><a id="user-content-making-your-code-citable-" class="anchor" aria-hidden="true" href="#making-your-code-citable-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Making your code citable <a name="user-content-Citation"></a></h3>
<p>If you want to make your code citable from the start, you should store the metadata needed for a citation from the start, by creating a <a href="https://codemeta.github.io" rel="nofollow">codemeta.json</a> file or a <a href="https://citation-file-format.github.io" rel="nofollow">CITATION.cff</a> file. Both will allow tooling that is currently being developed to automatically create citation information, rather than asking you to type it in a form later.</p>
<p>If you're interested, <a href="https://cite.research-software.org" rel="nofollow">cite.research-software.org</a> provides further background information about software citation in academia.</p>
<br>
<h2><a id="user-content-keeping-your-issues-up-to-date-" class="anchor" aria-hidden="true" href="#keeping-your-issues-up-to-date-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Keeping your issues up to date <a name="user-content-Issues"></a></h2>
<p>Issues are not necessarily problems with a project, but also suggestions for improvement, things to develop in the future, and comments and feedback about the project to work through. They can be openly shared and discussed with contributors as needed, sort of like a forum.</p>
<p>If you are a project lead, it is important to maintain a list of issues that make it clear to contributors what aspects of the project need attention. It is also important to engage with as many issues as possible from others in a positive manner, to show that you take their contributions seriously.</p>
<p>Key elements for issues include:</p>
<ul>
<li>An informative title and description;</li>
<li>Coloud-coded labels/tags to help categorise and filter;</li>
<li>Milestones to associate issues with specific features or project phases;</li>
<li>Assignees indicate who is responsible for working on an issue;</li>
<li>Comments for providing feedback.</li>
</ul>
<p align="center"><a target="_blank" rel="noopener noreferrer" href="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/blob/master/content_development/images/issues.png"><img src="https://github.com/OpenScienceMOOC/Module-5-Open-Research-Software-and-Open-Source/raw/master/content_development/images/issues.png" width="800" style="max-width:100%;"></a></p>
<p align="center"><i>The issue tracker for the Open Scholarship Strategy project</i></p>
<br>
<p>Within issues it is possible to use @ mentions to notify other contirbutors about the issue, and to get the right people engaged in an effective manner. GitHub has an internal system of notifications, just like Facebook or Twitter, and can also send emails to people who are mentioned in the issue tracker. This can all be customised for individuals within the user settings.</p>
<br>
<h2><a id="user-content-checklist-for-launching-your-project-" class="anchor" aria-hidden="true" href="#checklist-for-launching-your-project-"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Checklist for launching your project <a name="user-content-Checklist"></a></h2>
<p>So now you are ready to launch your project, begin advertising it, and getting contributions! Before continuing, make sure that you have:</p>
<ul class="contains-task-list">
<li class="task-list-item"><input type="checkbox" id="" disabled="" class="task-list-item-checkbox"> Project has a memorable and informative name</li>
<li class="task-list-item"><input type="checkbox" id="" disabled="" class="task-list-item-checkbox"> Project has a LICENSE file that is an exact copy of an Open Source license</li>
<li class="task-list-item"><input type="checkbox" id="" disabled="" class="task-list-item-checkbox"> Complete documentation including a README, CONTRIBUTING, and CODE_OF_CONDUCT files</li>
<li class="task-list-item"><input type="checkbox" id="" disabled="" class="task-list-item-checkbox"> Project has at least one clearly labelled issue</li>
<li class="task-list-item"><input type="checkbox" id="" disabled="" class="task-list-item-checkbox"> Any code included at this stage is clearly structured and annotated</li>
</ul>
<p><strong>CONGRATULATIONS!</strong></p>
<p>You have now launched an Open Source research project! Hopefully, from here on out, your work will act to benefit the wider community, forge new collaborations, and create new and fantastic opportunities for you all.</p>
<p>From now on, it is all up to you! Some advice is to:</p>
<ul>
<li>Write clean code;</li>
<li>Have a well-structured project;</li>
<li>Make frequent commits with clean messages;</li>
<li>Keep projects well-documented;</li>
<li>Have clear contributing guidelines;</li>
<li>Make use of the description and tag functions;</li>
<li>Don't fork someone else's repository unless you intend to work on them;</li>
<li>Make sure to contribute to other people's projects too.</li>
</ul>
